6リスク

tar.gz 展開時のパストラバーサル（severity: medium）
悪意ある ../../ を含む tar エントリを展開するリスク。starter-kit は自分で管理するため実害は低いが、外部テンプレートを許可する場合は要対処。関連: github.go / extractTemplate()
huh フォームの自動テストが困難（severity: low）
charmbracelet/huh はターミナルを必要とするため、prompt.go の動作確認は手動のみ。テストカバレッジに空白が生まれる。

7次に読むべき箇所（着手順）

  1. internal/template/source.go — インターフェース定義。全ての土台。
  2. internal/template/github.go の extractTemplate() — tar.gz 展開の核心ロジック。難易度 high。
  3. internal/template/local.go + testdata/ — テスト戦略を理解する。
  4. internal/cli/new.go — 全コンポーネントを繋ぐオーケストレーター。
  5. .github/workflows/release.yml — release-please → goreleaser の連携フロー。

8次に AI へ聞くとよい質問

テンプレート展開時に package.json の name フィールドをプロジェクト名に自動置換する場合、どこに処理を追加するのが最もクリーンか？
現設計では tar.gz を展開するだけで置換なし。v0.x で必要になった際の拡張ポイントを確認しておく価値がある。
GitHub の unauthenticated レート制限（60 req/h）に対して、将来的にトークン認証を追加する場合、GitHubSource のどのフィールドを拡張すればよいか？
現在は個人利用で問題ないが、設計の拡張ポイントを把握しておくと後で迷わない。
extractTemplate() でパストラバーサルを防ぐ最小限の追加チェックは？
セキュリティリスクの severity: medium。外部テンプレートを許可する前に対処が必要。
